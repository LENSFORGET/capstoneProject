# 小红书采集任务卡顿与超时原因分析

根据终端运行日志分析，采集任务运行极其缓慢且不断陷入死循环，主要由**浏览器自动化层**与**大模型 API 层**的双重“超时（Hang/Timeout）”叠加导致。

## 1. 浏览器自动化层（MCP工具连续 25s 超时）

**现象：**
- 尽管将 `navigate` 改为了后台异步执行（能瞬间返回），但在随后的工具调用中，无论是 `snapshot`、`type_text`，甚至是 `close` 和重新发起的 `navigate`，全部遭遇 25s 硬超时。
- 核心报错：`Error: 命令超时（25s），agent-browser 可能挂起。请手动关闭残留的 Chrome/agent-browser 进程后重试。`

**深层原因：**
1. **守护进程阻塞死锁（Daemon Blocked）：** 
   `agent-browser` (基于 Playwright) 运行时会启动一个后台守护进程处理会话。当我们用 `subprocess.Popen` 后台强行触发 `open` 命令后，守护进程正在全力拉取小红书页面（图片多、脚本复杂，加载耗时长）。此时我们立刻发起同步的 `snapshot` 请求，导致第二个进程视图连接或争夺这个尚未 Ready 的守护进程/页面实例，引发死锁或指令堆积，直到触发外部设置的 25s 超时。
2. **僵尸进程累积耗尽资源：** 
   由于上述死锁，MCP 超时并返回错误给 LLM。LLM 看到错误后尝试自救调用 `close()`。但由于守护进程已被卡死，`close()` 同样超时。这导致底层的无头 Chrome 进程和 Node.js 进程没有被杀掉。随着 LLM 的反复重试，系统中堆积了大量的僵尸进程，耗尽了内存与 CPU，最终彻底瘫痪。
3. **反爬与弹窗拦截：**
   小红书极其严格的风控可能会弹出滑块验证或强制登录弹窗，导致页面永远无法达到 `networkidle` 状态，DOM 结构也被破坏，从而引发 Playwright 内部操作无响应。

## 2. 大语言模型 API 层（长达 5 分钟的挂起）

**现象：**
- 日志显示 `14:29:03` NAT 框架向 NVIDIA 发起了模型推理请求。
- 直到整整 5 分钟后的 `14:34:05`，才抛出重试日志：`Retrying request to /chat/completions in 0.417746 seconds`。

**深层原因：**
1. **API 服务端延迟/限流（Rate Limit/Latency）：** 
   当前使用 `minimaxai/minimax-m2.1` 模型。由于请求频繁或者平台自身的接口拥挤，NVIDIA API 侧未响应数据，导致 HTTP 请求直接 Pending。
2. **缺乏合理的客户端短超时机制：** 
   NAT 的底层 OpenAI Client 没有设置较短的单次请求超时（例如 60s），导致程序傻等直到触发底层的 TCP 级别长超时（约 5 分钟）才进行 Retry。
3. **上下文激增带来的负担（Context Bloat）：** 
   由于前一个问题（浏览器工具不断报错），LLM 陷入了 "操作 -> 报错 -> 思考修复方案 -> 再操作" 的死循环。对话历史（History）变得非常长。每次请求都需要将这几万 Tokens 的无用报错历史传给服务器。过长的上下文（Long Context）成倍增加了服务器的首字生成时间（TTFT），进一步加剧了超时的概率。

---

## 结论与修复方向建议

当前卡顿是 **"不同步的浏览器页面状态"** 导致死锁，产生大量错误，这又催生了 **"超长上下文"**，最终将 **"不稳定的 API"** 彻底压垮，形成了恶性循环。

### 推荐解决步骤：
1. **恢复同步导航，放大初始超时（治本）：**
   废弃不稳定的后台 `Popen`。恢复同步的 `navigate`（或改用纯 Python 的 Playwright 直接驱动），并将导航允许的超时时间放宽至 60-90 秒。强制等待页面加载完毕再进行下一步，而不是盲目切后台引发并发抢占。
2. **增强环境彻底清理（硬核重启）：**
   在每次尝试 `close` 失败或发生超时错误时，执行严厉的系统级清理脚本（如 Windows 下的 `taskkill /F /IM chrome.exe` 和 `taskkill /F /IM node.exe`），斩断僵尸进程。
3. **切换更稳定的大模型及配置（优化 API）：**
   在 `workflow_scraper.yaml` 中将模型切换为响应更快、更稳定的模型（如 `meta/llama-3.1-70b-instruct` 或 `qwen2.5-72b-instruct`），并为其指定 `timeout: 60`，防止陷入 5 分钟级的无意义挂起。
